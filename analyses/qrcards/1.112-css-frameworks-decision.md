## Executive Summary

After comprehensive MPSE research (S1-S4 methodologies), the decision is to **continue using Bootstrap via CDN and skip CSS build tooling entirely**. This contradicts the initial impulse to adopt Tailwind or add Vite for CSS processing.

**Key Rationale**:
1. **Already working**: Bootstrap 4.5.2 CDN deployed, zero friction
2. **Form-heavy architecture**: Bootstrap's component library perfect for calculator UIs
3. **Solo developer efficiency**: CDN = zero build step, zero maintenance
4. **Strategic stability**: Bootstrap has highest viability score (9/10) - will outlast trends
5. **Vite decision independent**: Use Vite for JavaScript bundling, not CSS processing

## Why NOT Switch to Tailwind

### Research Findings

**S1 Rapid**: Tailwind wins popularity (27.7M downloads/week vs Bootstrap 4.9M)
**S2 Comprehensive**: Tailwind scores 89.5/100 (highest overall)
**S3 Need-Driven**: Tailwind excels for custom-designed UIs
**S4 Strategic**: Tailwind scores 8.5/10 viability (good, but lower than Bootstrap 9/10)

### But for QRCards...

❌ **Requires build step**:
- Need Vite to process Tailwind (add complexity)
- CDN version exists but loses tree-shaking benefit
- Purge CSS configuration needed (multidomain complexity)

❌ **Utility-first learning curve**:
- Solo dev needs to memorize utility classes
- Bootstrap semantic classes already familiar: `.form-control`, `.btn-primary`, `.card`
- Tailwind equivalent: `class="px-4 py-2 bg-blue-500 text-white rounded-lg hover:bg-blue-600"`

❌ **Calculator UIs = form-heavy**:
- Bootstrap's `.input-group`, `.input-group-append` perfect for calculator layouts
- Tailwind requires manual composition
- Bootstrap provides instant accessibility (ARIA attributes built-in)

❌ **Overkill for current needs**:
- QRCards calculators are functional, not brand-designed
- Bootstrap components = good enough
- Tailwind shines for marketing sites, custom brand systems (not QRCards use case)

✅ **Bootstrap advantages for QRCards**:
- Already deployed, zero migration cost
- Form components excellent (input groups, validation states)
- Server-template friendly (no build step)
- Strategic stability (9/10 viability, community-owned, 13+ years proven)

## Build Tool Strategy (1.114 + 1.112 Combined)

### JavaScript: Use Vite (from 1.114 decision)
- Bundle calculator JavaScript
- ES modules, TypeScript support
- Multidomain entry points
- HMR for development

### CSS: Skip Build Tools (Bootstrap CDN)
- NO Tailwind processing
- NO PostCSS pipeline
- NO CSS bundling

**Separation of Concerns**:
```
Vite handles:
  ✅ JavaScript bundling (calculator logic)
  ✅ Module resolution (imports)
  ✅ TypeScript compilation
  ✅ HMR for development

Bootstrap CDN handles:
  ✅ CSS framework
  ✅ Component styles
  ✅ Responsive grid
  ✅ Utility classes
```

**Why This Works**:
- Vite doesn't need to touch CSS (Bootstrap loads via CDN)
- JavaScript can still reference Bootstrap classes: `element.classList.add('btn-primary')`
- Zero CSS build complexity
- Fast development (no CSS rebuild needed)

## When to Reconsider (Future Migration Triggers)

### Migrate to Tailwind IF:
1. **Brand differentiation needed**: Custom design system required (not generic Bootstrap look)
2. **Bundle size critical**: Bootstrap CDN 150KB → Tailwind can purge to 10-20KB
3. **Design system maturity**: Ready to invest in custom component library
4. **Team growth**: Multiple designers/devs want utility-first workflow

### Upgrade Bootstrap IF:
1. **Bootstrap 5 features needed**: Offcanvas, improved forms, CSS variables
2. **Breaking changes acceptable**: Migrate templates from v4 to v5 (~4-8 hours)
3. **Accessibility improvements**: Bootstrap 5 has better WCAG 2.1 AA compliance

### Exit Criteria (Abandon Bootstrap):
- **Performance degradation**: 150KB CDN impacts page load on slow connections
- **Customization limits**: Need extensive theme customization (Bootstrap Sass build required)
- **Ecosystem abandonment**: Bootstrap maintenance stops (unlikely, 9/10 viability)
