## OAuth/OIDC Standard Decision (2.060 Research)

### Question: Should QRCards Adopt OAuth/OIDC Standard?

**Answer: NO** (Tier 3: Skip OAuth/OIDC)

### Why OAuth/OIDC Provides Negative ROI for QRCards

From `2.060-oauth-oidc/01-discovery/S3-need-driven/recommendation.md`:

**Use Case: Simple B2C SaaS MVP**
- Setup: 20-40 hours OAuth/OIDC vs 3-8 hours Firebase/Clerk/Supabase
- Migration savings: 10-20 hours per switch (not enough to justify)
- **Verdict: SKIP OAuth/OIDC**, use Firebase/Clerk/Supabase

**ROI Calculation:**
```
Setup cost: 30 hours × $200/hour = $6,000
Benefit: 10-20 hours saved per migration × P(migrate)
Expected migrations in 5 years: 1.0 (20% per year)
Expected benefit: 1.0 × 15 hours × $200 = $3,000
ROI: $3,000 / $6,000 = 0.5× return (NEGATIVE)
```

**Verdict: NOT JUSTIFIED (negative ROI)**

### OAuth/OIDC Value Sources (Ranked)

From 2.060 S3 synthesis:

1. **Enterprise SSO** (STRONG): $27,000-37,000 savings, unlocks $100K+ deals
   - ❌ QRCards: No enterprise customers requiring SAML SSO (yet)

2. **Multi-App SSO** (STRONG): $25,000 savings for 5-app ecosystem
   - ❌ QRCards: Single Flask app (no multi-app SSO needed)

3. **API Integrations** (MODERATE): Faster developer adoption
   - ❌ QRCards: Not an API platform (no third-party integrations)

4. **Portability** (WEAK): $3,600 expected value vs $6,000 setup cost (NEGATIVE ROI)
   - ❌ QRCards: Portability alone doesn't justify OAuth/OIDC

**Critical Insight from 2.060:**
> OAuth/OIDC is worth adopting for SSO use cases (multi-app or enterprise), NOT for portability. This is opposite of OpenTelemetry, where portability IS the core value proposition.

### Migration Effort Reality Check

**OAuth/OIDC Provider Switch:**
- Time: 80-150 hours (user data, admin APIs, MFA, templates)
- Portability: 15-30% (authorization flows only)
- What changes: User database, admin APIs, MFA, email templates
- Example: Auth0 → Keycloak in 100 hours

**Non-OAuth Provider Switch:**
- Time: 100-120 hours
- **Savings: Only 15-30 hours (NOT significant)**

**Compare to OpenTelemetry Backend Switch:**
- Time: 1-4 hours (change exporter endpoint)
- Portability: 95% (traces, metrics, logs APIs identical)
- **Savings: 38-146 hours (HIGHLY significant)**

**Verdict:** OAuth/OIDC portability is marginal (8-15% reduction in migration time), not worth 20-40 hour setup investment.

## Strategic Risks & Mitigation

### Risk #1: Supabase Acquisition (40% Probability by 2026-2028)

From 3.012 S4 strategic discovery:

> **Supabase Strategic**: "MEDIUM risk - acquisition target, auth monetization unclear"
> - Funding: $116M Series B (April 2023), $700M-1B valuation
> - Risk: Auth is 1 of 4 products (bundled pricing), unclear if standalone sustainable
> - Acquisition probability: 40% by 2026-2028 (Vercel, AWS, Microsoft targets)

**Mitigation Strategy:**
1. **Abstraction layer** (20-40 hours): Wrap Supabase Auth SDK in QRCards auth service interface
2. **Quarterly monitoring**: Track Supabase funding announcements, acquisition rumors, pricing changes
3. **Backup provider ready**: Test Clerk integration annually (8-20 hours), maintain migration runbook
4. **Self-hosting option**: If acquired with unfavorable terms, self-host Supabase on DigitalOcean/AWS (already open source)

**ROI of Abstraction Layer:**
- Investment: 20-40 hours ($4,000-8,000)
- Benefit: Reduce migration from 80-120 hours → 40-60 hours (50% reduction)
- Savings if migration needed: 40-60 hours ($8,000-12,000)
- Net value: $0-8,000 (break-even to positive)

**Verdict:** Build minimal abstraction layer (20-40 hours) to reduce lock-in risk.

### Risk #2: No Enterprise SSO (SAML) Support

**Limitation:** Supabase doesn't support enterprise SAML SSO (Auth0, Okta IdP integration)

**Impact:** If QRCards pivots to enterprise B2B requiring customer SSO, Supabase won't work.

**Mitigation:**
1. **WorkOS add-on**: Add WorkOS for SAML SSO ($125/connection) while keeping Supabase for core auth
2. **Hybrid approach**: Supabase for regular users (95% volume), WorkOS for enterprise SSO (5% enterprise customers)
3. **Migration path**: If enterprise becomes core, migrate Supabase → Auth0 (80-120 hours) when justified

**Cost Comparison (If Enterprise SSO Needed):**
- **Supabase + WorkOS**: $0 (Supabase) + $125/connection (WorkOS) = $625/month (5 connections)
- **Auth0 Enterprise**: $3,000-5,000/month (typical contract)
- **Savings**: $2,375-4,375/month ($28,500-52,500/year)

**Verdict:** Supabase + WorkOS hybrid is 75-85% cheaper than Auth0 Enterprise.

### Risk #3: RLS Complexity vs Application-Level Auth

**Challenge:** Supabase Row-Level Security (RLS) requires PostgreSQL policy knowledge.

**Setup complexity:**
- Application-level auth (Clerk/Auth0): Check `user.id === resource.owner_id` in application code
- RLS-based auth (Supabase): Define PostgreSQL policies: `CREATE POLICY customer_isolation ON trails FOR ALL USING (customer_id = auth.uid())`

**Learning curve:** 4-8 hours to understand RLS patterns, write policies, test isolation.

**Mitigation:**
1. **Start simple**: Use Supabase client libraries (RLS automatic for common patterns)
2. **Gradual adoption**: Begin with application-level checks, add RLS for defense-in-depth
3. **Documentation**: Supabase RLS examples extensive, QRCards use case well-documented

**ROI of RLS:**
- Investment: 4-8 hours learning RLS
- Benefit: Database-level isolation (vs application-level bugs causing data leakage)
- Security value: Defense in depth (if application code has bug, RLS still protects)
- Multi-tenant confidence: Can't accidentally query other customer's data

**Verdict:** RLS complexity is 4-8 hours one-time investment, worthwhile for multi-tenant security.

## Implementation Roadmap

### Phase 1: Setup Supabase Auth (Week 1, 5-8 hours)

**Tasks:**
1. Create Supabase project (10 min)
2. Configure email/password auth + Google OAuth (30 min)
3. Install Supabase client library (`pip install supabase-py`) (10 min)
4. Build minimal auth abstraction layer (3-4 hours):
   ```python
   # qrcards/auth/auth_service.py
   class AuthService:
       """Abstract auth interface (swappable providers)"""
       def sign_up(email: str, password: str) -> User:
           pass
       def sign_in(email: str, password: str) -> Session:
           pass
       def sign_out() -> None:
           pass
       def get_current_user() -> User | None:
           pass

   class SupabaseAuthService(AuthService):
       """Supabase implementation"""
       def __init__(self, supabase_client):
           self.client = supabase_client

       def sign_up(email: str, password: str) -> User:
           result = self.client.auth.sign_up({
               "email": email,
               "password": password
           })
           return self._map_user(result.user)
       # ... implement other methods
   ```
5. Add login/signup routes to Flask app (1-2 hours)
6. Test auth flows (signup, login, password reset, Google OAuth) (1-2 hours)

**Deliverables:**
- ✅ Supabase project configured
- ✅ Auth abstraction layer (provider-agnostic)
- ✅ Login/signup UI (basic forms)
- ✅ Google OAuth working
- ✅ User session management

### Phase 2: PostgreSQL Migration + RLS (Week 2-3, 12-20 hours)

**Prerequisites:** Complete 3.040 database services research (evaluate PostgreSQL migration)

**Tasks:**
1. Migrate SQLite schema to PostgreSQL (4-8 hours)
2. Define RLS policies for multi-tenant isolation (3-5 hours):
   ```sql
   -- Enable RLS on trails table
   ALTER TABLE trails ENABLE ROW LEVEL SECURITY;

   -- Policy: Users can only see trails they own
   CREATE POLICY customer_isolation ON trails
   FOR ALL
   USING (customer_id = auth.uid());
   ```
3. Test multi-tenant isolation (2-4 hours)
4. Migrate customer data to PostgreSQL (3-5 hours)

**Deliverables:**
- ✅ PostgreSQL database hosted on Supabase
- ✅ RLS policies enforcing customer isolation
- ✅ Existing customer data migrated
- ✅ Multi-tenant tests passing (Customer A can't see Customer B's trails)

### Phase 3: Customer Dashboard (Week 4-6, 20-30 hours)

1. Build customer dashboard (trail list, create new trail, edit trail) (10-15 hours)
2. Add role-based access control (admin vs viewer) (4-6 hours)

## Decision Summary

### Core Decision: Supabase Auth + SKIP OAuth/OIDC

**Why Supabase:**
1. ✅ Free to 50K MAU (vs Clerk $300/year, Auth0 $2,880/year)
2. ✅ PostgreSQL bundle (DB + Auth + Storage in one platform)
3. ✅ Open source & self-hostable (lock-in mitigation)
4. ✅ Row-Level Security (multi-tenant database isolation)
5. ✅ Simple use case fit (email/password + Google OAuth)

**Why Skip OAuth/OIDC:**
1. ❌ Negative ROI (-$1,600 over 5 years) for QRCards use case
2. ❌ No enterprise SSO requirement (yet)
3. ❌ No multi-app ecosystem (single Flask app)
4. ❌ Portability savings marginal (15-30 hours vs 20-40 hour setup cost)
5. ❌ Complexity not justified (20-40 hrs OAuth setup vs 5-8 hrs Supabase setup)

**Cost Comparison (5 Years):**
- **Supabase**: $1,000-1,600 total (5-8 hrs setup + $0/month)
- **Firebase**: $600-800 total (3-4 hrs setup + $0/month, but no PostgreSQL bundle)
- **Clerk**: $1,900-2,100 total (2-3 hrs setup + $300/year)
- **Auth0 OAuth/OIDC**: $18,400-22,400 total (20-40 hrs setup + $2,880/year)

**Savings: $800-21,400** vs alternatives

### Triggers for Re-evaluation

**Migrate FROM Supabase IF:**
- ❌ Acquisition with unfavorable terms (pricing +50%, platform lock-in)
- ❌ Enterprise customer requires SAML SSO (add WorkOS $125/connection OR migrate to Auth0)
- ❌ Service degradation (uptime <99%, support unresponsive)
- ❌ Need multi-app SSO (building separate admin portal, mobile app requiring unified auth)

**Add OAuth/OIDC IF:**
- ❌ Enterprise B2B pivot (Fortune 500 customers demanding SAML SSO)
- ❌ Multi-app ecosystem (3+ separate applications needing unified SSO)
- ❌ API platform pivot (developers integrating via OAuth 2.0 client credentials)

**Estimated probability:** 10-20% over next 3-5 years (QRCards likely stays simple B2C/SMB SaaS)
