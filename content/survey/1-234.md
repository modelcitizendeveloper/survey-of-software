## EXPLAINER - Quick Start

### What This Research Covers

Comprehensive analysis of bot SDK frameworks for building interactive chat bots across 5 major platforms: **Matrix** (matrix-nio), **Discord** (discord.py), **Telegram** (python-telegram-bot), **Nostr** (nostr-tools), and **Slack** (slack-bolt).

### Quick Decision Guide

| Platform | Choose if... | Avoid if... |
|----------|-------------|-------------|
| **Matrix** | Privacy/E2EE critical, self-hosting required | Need interactive UI (buttons/forms), quick MVP needed |
| **Discord** | Best UX/DX priority, gaming/community focus | Privacy required, payments needed, enterprise SSO |
| **Telegram** | Mobile-first, payments/commerce, fastest MVP | Rich formatting needed, threading critical |
| **Nostr** | Censorship resistance, Bitcoin/Lightning integration | Interactive UI needed, large user base required |
| **Slack** | Enterprise internal tool, workplace integration | Consumer bot, high message volume, budget constrained |

### Platform Comparison Summary

**Matrix (matrix-nio)**
- **Language**: Python
- **Best for**: Privacy-critical applications, self-hosting
- **Strengths**: E2EE, federated, open protocol
- **Weaknesses**: No native slash commands, complex interactive UI (widgets)
- **Time to MVP**: 4-8 hours

**Discord (discord.py)**
- **Language**: Python
- **Best for**: Gaming/community bots, best-in-class UX
- **Strengths**: Rich interactive components (Views), CommandTree, auto rate-limit handling
- **Weaknesses**: No E2EE, centralized, privileged intents approval for scale
- **Time to MVP**: 2-4 hours

**Telegram (python-telegram-bot)**
- **Language**: Python
- **Best for**: Mobile-first bots, payments, quick MVPs
- **Strengths**: Inline keyboards, payment API, 900M+ users, inline mode
- **Weaknesses**: No threading, 64-byte callback data limit
- **Time to MVP**: 2-4 hours

**Nostr (nostr-tools)**
- **Language**: JavaScript/TypeScript
- **Best for**: Censorship resistance, Bitcoin/Lightning integration
- **Strengths**: No approval process, Lightning Zaps, pseudonymous
- **Weaknesses**: No interactive UI, manual everything, relay unreliability
- **Time to MVP**: 4-8 hours

**Slack (slack-bolt)**
- **Language**: Python
- **Best for**: Enterprise internal tools
- **Strengths**: SSO, audit logs, Block Kit, Workflow Builder
- **Weaknesses**: Expensive ($8.75/user/mo), strict rate limits, not consumer-facing
- **Time to MVP**: 4-8 hours

### Quick Start Code Examples

#### Discord.py - Slash Command with Button
```python
import discord
from discord import app_commands

class Bot(discord.Client):
    def __init__(self):
        super().__init__(intents=discord.Intents.default())
        self.tree = app_commands.CommandTree(self)

client = Bot()

@client.tree.command(name="hello")
async def hello(interaction: discord.Interaction):
    await interaction.response.send_message("Hello!")

class ConfirmView(discord.ui.View):
    @discord.ui.button(label="Confirm", style=discord.ButtonStyle.success)
    async def confirm(self, interaction, button):
        await interaction.response.edit_message(content="Confirmed!", view=None)

client.run("TOKEN")
```

#### Telegram - Inline Keyboard
```python
from telegram import Update, InlineKeyboardButton, InlineKeyboardMarkup
from telegram.ext import Application, CommandHandler, CallbackQueryHandler

async def start(update: Update, context):
    keyboard = [[
        InlineKeyboardButton("Option 1", callback_data="1"),
        InlineKeyboardButton("Option 2", callback_data="2")
    ]]
    await update.message.reply_text("Choose:", reply_markup=InlineKeyboardMarkup(keyboard))

app = Application.builder().token("TOKEN").build()
app.add_handler(CommandHandler("start", start))
app.run_polling()
```

#### Matrix - E2EE Bot
```python
from nio import AsyncClient, RoomMessageText

client = AsyncClient("https://matrix.org", "@bot:matrix.org")

async def message_callback(room, event):
    if event.body.startswith("!hello"):
        await client.room_send(
            room.room_id,
            "m.room.message",
            {"msgtype": "m.text", "body": "Hello!"}
        )

client.add_event_callback(message_callback, RoomMessageText)
await client.login("password")
await client.sync_forever()
```

### Architecture Patterns

| Platform | Connection | Command Model | State Management | Error Handling |
|----------|-----------|---------------|-----------------|----------------|
| **Matrix** | Long-polling (sync) | Manual parsing | SQLite store (E2EE) | Response objects |
| **Discord** | WebSocket (gateway) | CommandTree (decorators) | Cache-based | Exceptions + auto-retry |
| **Telegram** | Polling/Webhook | Handler-based | Built-in persistence | Exceptions + manual retry |
| **Nostr** | WebSocket (multi-relay) | Manual parsing | Stateless (client) | Manual (no auto-retry) |
| **Slack** | Socket Mode/HTTP | Listener-based | External (Redis/DB) | Exceptions + auto-retry |

### Performance Deep Dive

#### Message Throughput
- **Discord**: 20-40 msg/s (50 req/s global limit, WebSocket advantages)
- **Telegram**: 10-20 msg/s (30 msg/s per chat, long-polling ~100-300ms)
- **Slack**: 1-3 msg/s (1 msg/s per channel - strictest limits)
- **Matrix**: 5-8 msg/s (10 req/s on matrix.org, long-polling ~200-500ms)
- **Nostr**: 10-50 msg/s (relay-dependent, no protocol limits)

#### Memory Footprint
- **Nostr**: ~10-15 MB (minimal, no caching)
- **Matrix**: 15-60 MB (depends on E2EE store size)
- **Telegram**: 20-30 MB (minimal with optional persistence)
- **Discord**: 30-50 MB minimal, **500MB-2GB** with full member cache (1000 guilds)
- **Slack**: 40-60 MB (includes SDK dependencies)

#### Rate Limiting Strategies
- **Discord**: Automatic pre-emptive rate limiting + queueing (transparent)
- **Slack**: Automatic retry on 429 + 5xx (transparent)
- **Telegram**: Manual implementation required (RetryAfter exception)
- **Matrix**: Manual implementation (RateLimitError response)
- **Nostr**: Relay-dependent (no standard protocol)

### Interactive Components Comparison

**Discord (Views)**:
- Class-based components
- Persistent views (survive bot restarts)
- Up to 5 buttons per row, 5 rows
- Modals with text inputs
- Autocomplete for slash commands

**Telegram (Inline Keyboards)**:
- Dict-based construction
- Up to 8 buttons per row, unlimited rows
- 64-byte callback data limit (use short IDs)
- Web Apps for complex UI
- No native modals

**Slack (Block Kit)**:
- JSON-based blocks
- Date pickers, time pickers, select menus
- Modals with validation
- Workflow Builder integration
- Most complex but most powerful

**Matrix (Widgets)**:
- Embed web content in messages
- Complex setup (host web UI, send widget event)
- Limited adoption (complexity barrier)
- Alternative: Use reactions + text commands

**Nostr (None)**:
- No interactive UI at protocol level
- Text + reactions (Kind 7 events) only
- Zaps (Lightning payments) for monetization

### E2EE Implementation Analysis

**Matrix (Full E2EE)**:
- Olm (1:1), Megolm (group) encryption
- Device verification + cross-signing
- Requires SqliteStore for persistent keys
- Performance: +100-200ms per message
- **Only platform with bot E2EE support**

**Telegram (Partial)**:
- Secret Chats (user-to-user E2EE only)
- Bots cannot create/use secret chats
- Regular chats: server-side encryption (Telegram can read)

**Nostr (Weak)**:
- NIP-04 (deprecated DM encryption)
- NIP-44 (improved DMs)
- Metadata not encrypted (sender, recipient, timestamp visible)
- No group encryption

**Discord, Slack (None)**:
- No E2EE support
- All messages readable by platform
- Enterprise Key Management (Slack) keeps keys in escrow (not true E2EE)

<summary><h2>S3: Need-Driven Analysis</h2></summary>

### Use Case Mapping

#### 1. Privacy-Critical Communication (Healthcare, Legal)
**Winner**: Matrix (matrix-nio)
- Only SDK with native E2EE for bot messages
- Self-hosting for data sovereignty
- Full message encryption including metadata
- Audit trail via homeserver logs

**Implementation notes**: Use SqliteStore, implement device verification, plan for key backup/recovery.

#### 2. High-Volume Notification Bot
**Winners**: Discord (20-40 msg/s) or Telegram (10-20 msg/s)
- Discord: Best for gaming/community (younger demographic)
- Telegram: Best for mobile-first (broader geographic reach, 900M+ users)
- Both free at scale (unlike Slack's per-user pricing)

**Avoid**: Slack (1-3 msg/s rate limits, $8.75/user/month)

#### 3. Enterprise Internal Tool
**Winner**: Slack (slack-bolt)
- SSO integration (SAML, SCIM)
- Admin audit logs
- Block Kit for rich forms/approvals
- Standard procurement process

**Alternative**: Matrix if privacy > UX (self-host, E2EE, but more IT overhead)

#### 4. Payments/Commerce Bot
**Winners**: Telegram (fiat) or Nostr (Bitcoin)
- Telegram: Native Stripe/Yandex.Money integration, in-app checkout
- Nostr: Lightning Zaps (NIP-57), Bitcoin-native monetization
- Discord: Payments explicitly banned in ToS (2024 update)

#### 5. Censorship-Resistant Bot
**Winner**: Nostr (nostr-tools)
- No central authority (relay network)
- Instant deployment (no approval)
- Pseudonymous (keypair-based identity)
- Lightning payments for monetization

**Alternative**: Matrix (federated, self-host for control, better UX than Nostr)

#### 6. Mobile-First Bot
**Winner**: Telegram (python-telegram-bot)
- 900M+ users primarily on mobile
- Excellent offline message queueing
- Inline keyboards optimized for mobile
- Push notifications native

#### 7. Developer Tool (CI/CD Integration)
**Winners**: Slack (teams) or Discord (OSS)
- Slack: Code blocks with syntax highlighting, threading, workplace integration
- Discord: Markdown code blocks, forum channels, large OSS community
- Both have excellent GitHub/GitLab webhook support

#### 8. Community Moderation Bot
**Winner**: Discord (discord.py)
- Best moderation API (ban, timeout, kick)
- AutoMod integration (regex-based filtering)
- Built-in audit logs
- Permission system (fine-grained role permissions)

### Decision Tree

```
START: What type of bot are you building?

Privacy-critical? → YES → Matrix (matrix-nio)
                 → NO  → Continue

Enterprise tool? → YES → Already use Slack? → Slack (slack-bolt)
                                           → Evaluate Matrix vs Slack
              → NO  → Continue

Payments? → YES → Bitcoin? → Nostr (nostr-tools)
                           → Telegram (python-telegram-bot)
         → NO  → Continue

Censorship resistance? → YES → Nostr (maximum) or Matrix (federated)
                      → NO  → Continue

High volume (>1000 msg/hr)? → YES → Gaming? → Discord (discord.py)
                                             → Telegram (python-telegram-bot)
                           → NO  → Continue

Mobile-first? → YES → Telegram (python-telegram-bot)
             → NO  → Continue

Developer tool? → YES → Internal? → Slack (slack-bolt)
                                 → OSS? → Discord (discord.py)
               → NO  → Continue

Default: Discord (discord.py) [best UX, most features]
```

### Anti-Patterns: When NOT to Use Each SDK

**Avoid Matrix if**: Interactive UI critical, quick MVP needed, non-technical users
**Avoid Discord if**: Privacy/E2EE required, payments needed, enterprise SSO
**Avoid Telegram if**: Rich formatting needed, threading critical, self-hosting required
**Avoid Nostr if**: Interactive UI needed, large user base required, UX polish matters
**Avoid Slack if**: Consumer-facing bot, high volume, budget-constrained (<$100/month)

### Market Segmentation (2026)

| Segment | Leaders | Growth (YoY) | Monetization | Outlook |

### Strategic Trends

#### 1. The "Bot Verification" Trap
Platforms increasingly gate bot features behind verification/approval:
- Discord: >75 guilds requires verification + privileged intents
- Slack: Enterprise Grid features require partnership
- Telegram: No restrictions (yet), but may change
- Matrix/Nostr: No approval (decentralized)

**Implication**: Early-stage bots should start on Telegram/Matrix (no gates), add Discord/Slack when proven.

#### 2. AI Agent Platforms
Platforms adding native AI features (Discord AI badge, Slack Workflow AI, Telegram enhancements). **Simple "answer questions" bots will be commoditized**. Differentiation moves to:
1. Domain expertise (vertical-specific bots)
2. Complex workflows (multi-step automation)
3. Privacy/security (where AI plugins can't go)

#### 3. Payment Platform Shift
- Telegram: Native payment API (Stripe, etc.)
- Discord: Commerce banned (2024 ToS)
- Nostr: Native Lightning (Zaps)
- Slack/Matrix: No native payments

**Long-term**: Payment rails = platform moat. Expect more platforms to add payments (or ban competitors).

#### 4. E2EE vs. Moderation Trade-off
Consumer platforms prioritizing moderation over E2EE (see Telegram's policy shifts). Enterprise platforms adding "compliant encryption" (keys in escrow).

**Implication**: Matrix remains only true E2EE option for bots. Moderation bots require server access (Discord/Slack best APIs).

### Technology Stack Recommendations

**Language Support**:
- **Python**: discord.py, python-telegram-bot, matrix-nio, slack-bolt (4/5 platforms) ✅
- **JavaScript/TypeScript**: discord.js, telegraf, matrix-bot-sdk, @slack/bolt (4/5 platforms) ✅
- **Go**: discordgo, gotgbot, mautrix-go (3/5 platforms)
- **Rust**: serenity, teloxide, matrix-rust-sdk (3/5 platforms)

**Recommendation**: Python or TypeScript for widest SDK support. Go/Rust for performance-critical (>10k msg/s).

### Deployment Patterns

**Single-Server Monolith** (MVP, <100 users):
- Bot process → Platform API → SQLite DB
- Cost: $5-10/month (VPS)
- All platforms work well

**Serverless Functions** (sporadic traffic):
- Webhook → AWS Lambda/Cloudflare Workers → Platform API
- Best: Slack, Telegram (webhook-friendly)
- Avoid: Discord (WebSocket), Matrix (long-polling), Nostr (WebSocket)
- Cost: $0-5/month

**Kubernetes Cluster** (HA, >10k users):
- Load balancer → N bot pods → Shared state (Redis/PostgreSQL)
- Best: Slack, Telegram (HTTP mode, stateless)
- Challenges: Matrix (E2EE state replication), Nostr (duplicate events)
- Cost: $50-500/month

### Risk Assessment (2026-2031)

| Platform | Shutdown Risk | API Breakage | Policy Tightening |
|----------|---------------|--------------|-------------------|
| Slack | Very Low | Low | Medium (anti-spam) |
| Discord | Very Low | Medium | **High** (verification gates) |
| Telegram | Low | Low | Medium (may add restrictions) |
| Matrix | Very Low (protocol) | Medium | Very Low |
| Nostr | None (protocol) | **High** (NIPs evolving) | None |

**Mitigation**: Abstract platform logic (adapter pattern), monitor changelogs, test beta APIs, multi-platform redundancy.

### 5-Year Outlook

**Predicted consolidation**:
- Slack partners with Microsoft (Teams integration)
- Discord explores federation (Matrix-compatible)
- Telegram continues independent (founder-funded)
- Matrix reaches 10M+ MAU (critical mass)
- Nostr remains niche (Bitcoin community)

**Technology shifts**:
- Platform interoperability increases (Matrix bridges standardize)
- Bot verification becomes universal (anti-spam)
- AI features commoditize simple bots (need specialization)
- WebAssembly bots (speculative): platform-agnostic binaries

### Emerging Opportunities

1. **AI-Augmented Moderation**: LLM-based content analysis + platform mod APIs (Discord/Telegram best)
2. **Cross-Platform Identity**: Link Discord/Telegram/Matrix accounts, unified profile
3. **Web3 Integration**: Wallet verification, DAO governance (Discord/Nostr)
4. **Compliance/Audit Bots**: Message archiving, keyword alerting (Slack/Matrix for enterprises)

## Key Takeaways

1. **No single "best" platform** - optimize for use case (privacy → Matrix, UX → Discord, mobile → Telegram, enterprise → Slack, censorship-resistance → Nostr)

2. **Platform risk is real** - abstract platform logic early (adapter pattern), monitor changelogs, plan for multi-platform when proven

3. **Architecture patterns**:
   - Thin client (webhooks): Slack, Discord, Telegram
   - Stateful workflows: Telegram (built-in), Discord (persistent Views)
   - Real-time sync: Discord, Slack (message editing)
   - Collaborative: Slack, Discord (threading, rich UI)

4. **Cost analysis**: Telegram/Discord/Nostr free at scale, Matrix self-host costs, Slack $8.75/user/month (100x more expensive)

5. **Future-proofing**: AI commoditizes simple bots, differentiate via domain expertise, complex workflows, or privacy/security
